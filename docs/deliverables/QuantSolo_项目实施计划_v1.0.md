## §1 项目总体说明

### 1.1 项目目标

**目标一（最重要）**：3 个月内完成模拟盘全链路上线（数据→信号→风控→模拟执行→对账→告警闭环），开始累积 B 类行为数据。

**目标二**：依据 QS-CAL-001 统计闸门校准结论（闸门编号引用，数字不重列），经 B1+B3 闸门评估后，最早 M9 将实盘资金从 5 万扩至 20 万。

**底线**：任何阶段不得在统计闸门未通过的情况下上调实盘资金规模；25% 硬止损不可妥协（铁律四，见 QS-C01 §1）。

### 1.2 单人项目约束

| 约束 | 数值 | 说明 |
|------|------|------|
| 周工时中位数 | **15h/周** | 业余兼职，最高 20h/周，不可持续压榨 |
| 月工时预算 | **60–75h/月**（4–5 周按 15h 计） | 超支 20% = 72–90h 触发砍范围 |
| 3 个月总预算 | **约 210h**（M1:60 + M2:90 + M3:60）| 见 baseline_spec.md §7 |
| 研究/运维分配 | **70% / 30%** | 超出 30% 运维即触发简化评审 |
| 总运维上限 | **≤4h/周** | 超出则砍范围，不增加人力 |

### 1.3 上线定义澄清

> **「M3 末上线可用」≠「满仓实盘」**（依据 QS-C01 §16.1 及 QS-CAL-001 §四）。

| 阶段 | 时点 | 上线的含义 |
|------|------|---------|
| M3 末 | 第3月末 | 模拟盘全链路上线，**开始累积** B 类行为数据 |
| M4 起 | M3+4 周模拟零差错后 | 5 万（A2 降级则 2.5 万）**真实实盘**启动 |
| 最早 M9 | 满足 B1+B3 闸门且 ≥13 周真实实盘 | **加仓**：5 万→20 万 |

26 周观测窗是统计功效决定的物理下限（QS-CAL-001 §三），不可用工期压缩。

## §3 关键路径与依赖关系

### 3.1 关键路径说明

项目关键路径（Critical Path）如下，任何环节延误直接影响 M3 末上线目标：

```
[国金开户+入金10万] ──→ [miniQMT 计时3个月] ──→ [M3末前xtquant就绪]
        │
[M1 数据管道上线] ──→ [pytest 13场景全绿] ──→ [M2 因子研究可开始]
        │
[M2 线性基线夏普≥0.6] ──→ [test A1/A2 通过] ──→ [M3 模拟盘可启动]
        │
[M2 执行架构+状态机] ──→ [M3 QMT模拟对接] ──→ [M3 模拟盘全链路]
        │
[M3 连续4周零差错] ──→ [合规报备完成] ──→ [M4 小实盘启动]
```

### 3.2 关键依赖矩阵

| 任务 | 依赖于 | 依赖类型 | 延误影响 |
|------|--------|---------|---------|
| 因子研究开始（M2） | pytest 13 场景全绿（M1 末） | 强依赖 | M2 推迟 |
| 执行层 xtquant 开发（M2 W8） | 国金账户 + miniQMT 可用 | 软依赖（可先 Mock 开发）| xtquant 集成推迟 |
| test 段 A1/A2 终评（M2 W7） | 线性基线夏普 ≥0.6（M2 W5）| 强依赖 | test 评估延后 |
| QMT 模拟盘接入（M3 W9） | 执行三层架构完成（M2 W8）| 强依赖 | M3 整体推迟 |
| M4 小实盘启动 | 模拟盘连续 4 周零差错 + 合规报备完成 | 双强依赖 | M4 推迟 |
| M9 加仓评估 | B1+B3 闸门：26 周累计观测（含 ≥13 周真实实盘）| 物理时间下限 | 最早 M9，不可压缩 |

### 3.3 国金开户关键路径（M1 第1周必须启动）

> **理由**：国金证券 miniQMT 需要入金 10 万保持 3 个月后才可永久开通（QS-C01 §10.2）。若 M1 第1周不启动，3 个月计时无法在 M3 末前完成，直接阻塞 M3 模拟盘上线。

| 步骤 | 时间点 | 动作 | 验收 |
|------|--------|------|------|
| 开户申请 | M1 W1 Day1–Day3 | 国金证券官网/APP 开户，勾选程序化交易 | 开户申请提交 |
| 入金 10 万 | M1 W1 Day3–Day5 | 银行转账至国金账户，触发 miniQMT 3 个月计时 | 资金到位确认 |
| miniQMT 权限申请 | M1 W1 Day5 | APP 内申请 QMT+miniQMT，联系客户经理，风测选激进型 | 申请已提交 |
| 下载官方客户端 | M1 W2 | 从短信链接下载（务必官方版，版本错会报错）| 客户端安装成功 |
| xtquant 获取 | M1 W2 | 从 QMT 客户端「下载 Python 库」，拷贝至 site-packages | Python import 成功 |

### 10.2 砍范围序列（按顺序执行，不可跳步）

#### 第一刀：砍 LLM 舆情至 M4+

**触发条件**：任一月份超支 20%，或 LLM 离线 IC 验证（M1 W4）未通过四条红线。

**砍的内容**：
- M1 W4 的 LLM IC 验证结果决定路线：不通过则直接砍，不等超支触发；
- 停止 LLM 舆情数据采集与处理管道开发；
- 停止舆情 IC 验证红线后续任务（第五条红线）；
- 推迟至 M4 后视余力重新评估。

**不受影响的内容**：
- 主策略（多因子 + 趋势卫星 + 择时）不受影响；
- 线性基线 + LightGBM（如未触发第二刀）继续；
- 上线时间表不受影响。

**影响**：差异化 Alpha 路线延后，策略暂为「无舆情」版本。

---

#### 第二刀：砍 LightGBM，保留线性基线

**触发条件**：砍第一刀后仍超支 20%（即月工时仍超 72h/84h）。

**砍的内容**：
- 停止 LightGBM 模型训练与验证；
- 停止 LightGBM 增量验收评估；
- 上线策略降级为纯线性 Ridge 多因子（夏普 0.6 可放行研究闸门）。

**不受影响的内容**：
- 线性基线（夏普 ≥0.6 → 满足 A1/A2 研究判定）；
- 风控/执行/监控/对账全部不受影响；
- 上线可行，只是 Alpha 源减少一层非线性增强。

**注意**：**线性基线夏普 0.6 满足研究放行门槛，但上实盘硬线是夏普 0.8**。若砍 LightGBM 后线性基线夏普在 0.6–0.8 之间，需记录「低于上实盘硬线 0.8」的风险，并在 M4 前重新评估（可选择 M3 延长模拟盘阶段补强）。

---

#### 第三刀：M3 模拟盘延长至第4个月

**触发条件**：砍前两刀后工期仍超支，或 M3 验收未通过（QS-C05 清单有未通过项）。

**砍的内容**：
- M3 上线日期推迟 2–4 周；
- M4 实盘相应后移，B1 观测窗起始点后移。

**不受影响的内容**：
- 研究质量不降低；
- 风控标准不降低；
- 最终上线系统的功能与质量不妥协。

### 12.4 节奏保护原则

> 一人公司最容易失败的两种节奏：**过度努力（burn out）** 和 **间歇性放弃（逃避复盘）**。

**防 burn out**：
- 明确运维上限 ≤4h/周；超出即触发砍范围，不靠堆时间；
- M2 峰值工时（22.5h/周）只允许持续 4 周；M3 回落至 15h/周；
- 任务最小化分解：每周 ≤3 个核心任务，完成即有成就感。

**防逃避复盘**：
- 周度对账**不可取消**（铁律五，见 QS-C01 §1）；
- 即使本周没有亏损/没有告警，对账复盘仍需执行；
- 对账差异是早期预警信号，不等到账户亏损才发现。

**防进度焦虑**：
- 关键路径延误 ≤2 周不是问题，及时激活砍范围应对；
- B1/B2 判线数字是统计上严格的物理下限，不可焦虑性加速；
- 前两年 ROI 大概率为负是系统规划的一部分，不是失败信号。
